## Requirements specification for Formloop

### Functional requirements

| ID       | Requirement                                                                                                                                                                              | Rationale                                                                                     | Status   |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------- | -------- |
| FL-F-001 | Formloop shall provide a chat-based interface for submitting design requests and revisions.                                                                                              | Chat is the primary human interaction surface.                                                | Proposed |
| FL-F-002 | Formloop shall maintain a normalized current fit/form/function specification for the active design.                                                                                      | The current spec is a core user-facing and agent-facing object in the agreed design.          | Proposed |
| FL-F-003 | Formloop shall provide a manager agent that coordinates the active run and delegates to specialist agents.                                                                               | The agreed architecture uses a manager-plus-specialists harness.                              | Proposed |
| FL-F-004 | Formloop shall provide at minimum the specialist roles CAD Designer, Design Researcher, Render Specialist, Review Specialist, and Eval Specialist.                                       | These are the agreed initial specialist boundaries.                                           | Proposed |
| FL-F-005 | Formloop shall support an internal review loop for normal design runs where no ground-truth geometry is available.                                                                       | The project explicitly distinguishes internal design-loop review from developer evals.        | Proposed |
| FL-F-006 | Formloop’s internal review loop shall assess the latest candidate against the normalized spec, rendered outputs, deterministic inspections, and optional user-provided reference images. | This is the agreed basis for closed-loop design review without ground truth.                  | Proposed |
| FL-F-007 | Formloop shall support deterministic measurement and inspection requests inside the internal review loop.                                                                                | Internal review requires concrete checks, not only visual judgment.                           | Proposed |
| FL-F-008 | Formloop shall support reference-image review as a closed-loop review tool when the user provides a reference image.                                                                     | Reference-image comparison is an explicitly requested review capability.                      | Proposed |
| FL-F-009 | Formloop shall allow the Review Specialist to request further revision from the CAD Designer when the current candidate is not acceptable.                                               | The review loop must be able to drive iterative improvement.                                  | Proposed |
| FL-F-010 | Formloop shall expose the latest geometry previews to the user, including the standard multi-view render outputs.                                                                        | Users need direct visibility into the latest candidate geometry.                              | Proposed |
| FL-F-011 | Formloop shall expose a concise latest review summary to the user.                                                                                                                       | The UI should show the state of the internal review loop without overwhelming detail.         | Proposed |
| FL-F-012 | Formloop shall expose downloadable artifacts including STEP, GLB, render sheet, and model source when available.                                                                         | Artifact download is part of the core user flow.                                              | Proposed |
| FL-F-013 | Formloop shall provide expandable access to tool-call history, subagent-call history, and intermediate traces.                                                                           | Traceability is required, but secondary details should stay out of the primary UI by default. | Proposed |
| FL-F-014 | Formloop shall support a high-level operator CLI for application lifecycle management.                                                                                                   | The architecture explicitly defines a separate Formloop operator CLI.                         | Proposed |
| FL-F-015 | Formloop’s operator CLI shall support UI start, stop, and status operations.                                                                                                             | Application lifecycle management is part of the agreed boundary for Formloop.                 | Proposed |
| FL-F-016 | Formloop’s operator CLI shall support single-run execution outside the UI.                                                                                                               | The system should support individual queries outside the interactive app surface.             | Proposed |
| FL-F-017 | Formloop shall support developer eval runs over one or more datasets with known ground-truth geometry.                                                                                   | Developer evals are a first-class project capability.                                         | Proposed |
| FL-F-018 | Formloop evals shall combine deterministic calculators and agent eval judges.                                                                                                            | The project explicitly requires both objective metrics and higher-level judge assessments.    | Proposed |
| FL-F-019 | Formloop evals shall support tool-using agent judges where useful for multi-step assessments such as dimensional compliance.                                                             | The user explicitly allowed agentic judges in eval routines.                                  | Proposed |
| FL-F-020 | Formloop evals shall generate per-case outputs and aggregate batch outputs.                                                                                                              | Both detailed failure analysis and regression summaries are required.                         | Proposed |
| FL-F-021 | Formloop shall support CI-triggered batch eval execution and reporting.                                                                                                                  | CI regression detection is part of the agreed evaluation model.                               | Proposed |
| FL-F-022 | Formloop shall manage standardized artifact types across design runs, internal review runs, and eval runs.                                                                               | Shared artifact conventions are necessary for UI, tooling, and CI interoperability.           | Proposed |
| FL-F-023 | Formloop shall support a skill system that provides reusable operational knowledge to agents.                                                                                            | Skills are a central design element for keeping prompts cleaner and more reusable.            | Proposed |
| FL-F-024 | Formloop shall allow the manager agent to request external research when required to complete ambiguous or under-specified designs.                                                      | The Design Researcher role exists specifically to fill this gap.                              | Proposed |
| FL-F-025 | Formloop shall keep manufacturing or slicing validation outside the core design loop in v1.                                                                                              | The project explicitly treats 3D printing validation as optional future scope.                | Proposed |

### Non-functional requirements

| ID        | Requirement                                                                                                                      | Rationale                                                                                   | Status   |
| --------- | -------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------- | -------- |
| FL-NF-001 | Formloop shall prioritize safe and reliable execution over maximal autonomy.                                                     | Safety and reliability are explicit project objectives.                                     | Proposed |
| FL-NF-002 | Formloop shall keep the primary user interface intuitive and low-noise by default.                                               | The agreed UI philosophy emphasizes clarity and avoids exposing excess detail.              | Proposed |
| FL-NF-003 | Formloop shall provide intermediate progress updates during multi-step runs.                                                     | The user explicitly wants visible progress during agent work.                               | Proposed |
| FL-NF-004 | Formloop shall preserve artifact traceability across runs, revisions, review steps, and eval results.                            | Traceability is a cross-cutting design objective.                                           | Proposed |
| FL-NF-005 | Formloop shall support reproducible eval execution for regression tracking.                                                      | Developer evals must be usable as a stable quality bar over time.                           | Proposed |
| FL-NF-006 | Formloop shall be operable in a headless environment through its CLI and CI integrations even if the UI is not running.          | Automation and CI must not depend on an interactive UI session.                             | Proposed |
| FL-NF-007 | Formloop shall keep the internal design loop and developer eval loop conceptually distinct in both implementation and reporting. | The user explicitly requested clarity between these two modes.                              | Proposed |
| FL-NF-008 | Formloop shall expose enough logs and structured data to debug failures in design runs and eval runs.                            | High-quality iteration depends on diagnosability.                                           | Proposed |
| FL-NF-009 | Formloop should minimize unnecessary context growth by using specialist boundaries and skills intentionally.                     | Context cleanliness is one of the explicit reasons for the subagent and skill architecture. | Proposed |
| FL-NF-010 | Formloop shall support CI reporting on push to main and a smaller smoke path for PRs.                                            | This is part of the agreed developer workflow.                                              | Proposed |

### Design and technical constraint requirements

| ID       | Requirement                                                                                                                                | Rationale                                                                                          | Status   |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------- | -------- |
| FL-D-001 | Formloop shall use `cad-cli` as its deterministic CAD tool substrate rather than embedding those responsibilities directly in the app.     | The architecture deliberately separates deterministic tool execution from the application harness. | Proposed |
| FL-D-002 | Formloop shall use build123d as the primary modeling backend exposed through its design workflows.                                         | This is a core technical decision for the project.                                                 | Proposed |
| FL-D-003 | Formloop shall use Blender-rendered GLB outputs as the standard preview path presented to users.                                           | This is the agreed rendering contract.                                                             | Proposed |
| FL-D-004 | Formloop shall treat STEP as the authoritative geometry artifact in its internal state and eval workflows.                                 | STEP is the agreed source-of-truth artifact for CAD and comparison.                                | Proposed |
| FL-D-005 | Formloop shall expose a centralized internal runtime abstraction for CLI execution, Python execution, artifact reads, and artifact writes. | The agreed harness uses a constrained runtime rather than a broker service.                        | Proposed |
| FL-D-006 | Formloop shall not require a separate tool broker service in v1.                                                                           | The user explicitly rejected a broker-based design for the initial architecture.                   | Proposed |
| FL-D-007 | Formloop shall represent the current design state explicitly rather than reconstructing it from raw chat alone.                            | The UI and manager both depend on a normalized current state object.                               | Proposed |
| FL-D-008 | Formloop shall keep internal review artifacts and eval artifacts distinct, even when some underlying tools are shared.                     | The project requires clarity between internal review and developer evaluation.                     | Proposed |
| FL-D-009 | Formloop shall support standardized output schemas for deterministic metrics and judge outputs.                                            | Evals require consistent aggregation and CI reporting.                                             | Proposed |
| FL-D-010 | Formloop shall reserve an optional future integration point for printability validation but shall not make it a core release requirement.  | 3D printing validation is intentionally deferred from the core project boundary.                   | Proposed |

These two requirement sets are meant to be a working baseline. The most important maintenance rule is simple: **keep the Status column current as development progresses**, so the spec remains a living control document rather than a one-time design artifact.
